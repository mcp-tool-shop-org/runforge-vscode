## v1.1.0 में क्या नया है

1. **चल रहे प्रशिक्षण को रद्द करें** (`RunForge: Cancel Active Training`) - कमांड पैलेट या VS Code के प्रगति-अधिसूचना रद्द बटन के माध्यम से चल रहे प्रशिक्षण को रद्द करें। 5 सेकंड का अनुग्रहकालीन SIGTERM विंडो, फिर SIGKILL। रद्द किए गए रन में `.cancelled` मार्कर जोड़ा जाता है ताकि पुनर्प्राप्ति और रन पिकर उन्हें सही ढंग से वर्गीकृत कर सकें।
2. **इंडेक्स को पुनर्प्राप्त करें** (`RunForge: Recover Index`) - `.ml/runs/` में मौजूद सभी रन को `.ml/outputs/index.json` में फिर से जोड़ता है। यह प्रक्रिया idempotent है। यह क्रैश होने वाले लेखन या कार्यक्षेत्र को स्थानांतरित करने के बाद उपयोगी है।
3. **कार्यक्षेत्र सुरक्षा** - Python सबप्रोसेस अब `vscode.workspace.isTrusted` की आवश्यकता होती है। अविश्वसनीय कार्यक्षेत्रों को एक कार्रवाई योग्य SafeError प्राप्त होता है जो Manage Workspace Trust UI की ओर इशारा करता है।
4. **प्रति-युग प्रगति सूचनाएं** - प्रशिक्षण लाइव प्रगति दिखाता है और `vscode.window.withProgress` के माध्यम से एक रद्द बटन प्रदर्शित करता है।
5. **बेहतर CSV त्रुटि संदेश** - गैर-कॉमा विभाजक, गैर-UTF-8 एन्कोडिंग, सभी NaN लेबल, एकल-कॉलम CSV और केवल हेडर वाले CSV, प्रत्येक के लिए विशिष्ट, कार्रवाई योग्य निदान प्रदर्शित होते हैं, बजाय अस्पष्ट pandas ट्रेसबैक के।
6. **कस्टम ESLint नियम** - [`docs/CONTRACTS.md`](docs/CONTRACTS.md) में कोडित वास्तुशिल्प सिद्धांतों को लागू किया जाता है (कोई भी canonical-value लिटरल दोहराव नहीं, उपभोक्ता मॉड्यूल में कोई भी shadow प्रकार नहीं)।
7. **दस्तावेज़** - [`docs/CONTRACTS.md`](docs/CONTRACTS.md) अब छह वास्तुशिल्प नियमों + पांच चरणों के संरचित ऑडिट से सात परिचालन पैटर्न को कोडित करता है। क्रॉस-डोमेन (TS / Python / अवलोकन) कार्य के लिए ये पैटर्न अपरिवर्तनीय हैं।

## 🛡️ रनफ़ॉर्ज गारंटी

रनफ़ॉर्ज एक ऐसा सॉफ्टवेयर है जो "यह मेरे मशीन पर काम करता है" को फोरेंसिक निश्चितता से बदलने के लिए डिज़ाइन किया गया है।

### हम क्या गारंटी देते हैं
1. **नियतिवाद**: प्रत्येक रन को एक बीज दिया जाता है। समान डेटा के साथ समान प्रीसेट को समान बीज के साथ फिर से चलाने से बिल्कुल वही मॉडल प्राप्त होता है।
2. **उत्पत्ति**: प्रत्येक `run.json` रिकॉर्ड में Git Commit SHA, Python इंटरप्रेटर पथ और उपयोग किए गए एक्सटेंशन संस्करण शामिल हैं। आप किसी भी मॉडल को उस कोड तक वापस ट्रैक कर सकते हैं जिसने इसे बनाया था।
3. **जांच क्षमता**: कलाकृतियां (मॉडल, मेट्रिक्स, लॉग) मानक प्रारूपों (JSON, joblib) में डिस्क पर सहेजी जाती हैं। कोई भी छिपी हुई डेटाबेस नहीं, कोई भी क्लाउड निर्भरता नहीं।

### यह क्या नहीं है
- **कोई जादू ऑटोएमएल टूल नहीं**: हम यह अनुमान नहीं लगाते कि आप क्या चाहते हैं। हम विशिष्ट, ट्यून करने योग्य प्रीसेट चलाते हैं।
- **कोई क्लाउड प्लेटफॉर्म नहीं**: हम आपके डेटा को कहीं भी नहीं भेजते हैं। सब कुछ आपके VS Code कार्यक्षेत्र में स्थानीय रूप से होता है।

पूर्ण विश्वास मॉडल के लिए, [docs/TRUST_MODEL.md](docs/TRUST_MODEL.md) देखें।

### सुरक्षा और डेटा दायरा

**डेटा तक पहुंच:** वर्कस्पेस CSV फाइलें (प्रशिक्षण के लिए केवल-पढ़ने योग्य), `.ml/` डायरेक्टरी (रन मेटाडेटा, मॉडल आर्टिफैक्ट, मेट्रिक्स JSON), पायथन सबप्रोसेस का stdout/stderr। **डेटा तक नहीं पहुंच:** वर्कस्पेस के बाहर की कोई भी फाइल, कोई भी ब्राउज़र डेटा, कोई भी ऑपरेटिंग सिस्टम क्रेडेंशियल। **आवश्यक अनुमतियाँ:** केवल वर्कस्पेस के भीतर फाइल सिस्टम पढ़ने/लिखने की अनुमति, पायथन सबप्रोसेस निष्पादन। **कोई नेटवर्क आउटगोइंग नहीं:** सभी ऑपरेशन स्थानीय हैं। **कोई टेलीमेट्री** एकत्र या भेजा नहीं जाता है।

### रन का जीवनचक्र

```
dataset.csv
    │
    ▼
┌─────────────────────────────────────────────────────────────┐
│  Training (run_training)                                    │
│                                                             │
│  1. Validate dataset (label column, numeric values)         │
│  2. Compute dataset fingerprint (SHA-256)                   │
│  3. Split 80/20 train/val (deterministic, stratified)       │
│  4. Fit pipeline (StandardScaler + Classifier)              │
│  5. Compute metrics                                         │
│  6. Extract interpretability (if supported)                 │
└─────────────────────────────────────────────────────────────┘
    │
    ▼
.ml/runs/<run-id>/
    ├── run.json                              ← Metadata + pointers
    ├── metrics.json                          ← Phase 2 metrics (3 keys)
    ├── metrics.v1.json                       ← Detailed metrics by profile
    └── artifacts/
        ├── model.pkl                         ← Trained pipeline
        ├── feature_importance.v1.json        ← (RandomForest only)
        ├── linear_coefficients.v1.json       ← (Linear models only)
        └── interpretability.index.v1.json    ← Unified index
```

## गारंटी (v0.2.1+)

RunForge VS Code, नियतात्मक, अनुबंध-आधारित मशीन लर्निंग प्रशिक्षण प्रदान करता है। नीचे दी गई गारंटी जानबूझकर हैं और परीक्षणों द्वारा लागू की जाती हैं।

### नियतात्मकता

समान डेटासेट, कॉन्फ़िगरेशन और RunForge संस्करण को देखते हुए:

- प्रशिक्षण/सत्यापन विभाजन सभी रन में समान होते हैं।
- उत्पन्न आर्टिफैक्ट पुनरुत्पादित होते हैं।
- मेट्रिक्स आउटपुट स्थिर होते हैं।

स्पष्ट रूप से सीड किए गए व्यवहार के बाहर कोई यादृच्छिकता नहीं है।

### लेबल हैंडलिंग

- लेबल कॉलम को स्पष्ट रूप से निर्दिष्ट किया गया है।
- लेबल कॉलम की स्थिति के आधार पर अनुमानित नहीं किया जाता है।
- गलत तरीके से कॉन्फ़िगर किए गए या गुम लेबल शुरुआती चरण में विफल हो जाते हैं।

### मेट्रिक्स अनुबंध

प्रशिक्षण बिल्कुल तीन मेट्रिक्स उत्पन्न करता है:

```json
{
  "accuracy": number,
  "num_samples": number,
  "num_features": number
}
```

कोई अतिरिक्त फ़ील्ड निहित रूप से नहीं जोड़ा जाता है।
स्कीमा विस्तार के लिए एक संस्करणित अनुबंध परिवर्तन की आवश्यकता होती है।

### मॉडल आर्टिफैक्ट

- `model.pkl` हमेशा एक क्रमबद्ध `sklearn.Pipeline` होता है।
- सभी प्रीप्रोसेसिंग (जैसे स्केलिंग) एम्बेडेड है।
- आर्टिफैक्ट आत्मनिर्भर है और अनुमान लगाने के लिए तैयार है।

किसी भी बाहरी प्रीप्रोसेसिंग चरणों की आवश्यकता नहीं है।

### गुम डेटा

- गुम मान वाले पंक्तियों को नियतात्मक रूप से हटा दिया जाता है।
- हटाए गए पंक्तियों की संख्या लॉग की जाती है।
- कोई भी गुप्त प्रतिस्थापन नहीं होता है।

### सत्य का स्रोत

- सभी पायथन निष्पादन तर्क `python/ml_runner/` में मौजूद हैं।
- कोई डुप्लिकेट या छाया कार्यान्वयन नहीं है।
- TypeScript और पायथन व्यवहार के बीच समानता को परीक्षण लागू करते हैं।

### स्थिरता नीति

- v0.2.1 पर व्यवहार को स्थिर किया गया है।
- ब्रेकिंग परिवर्तन के लिए एक स्पष्ट प्रमुख संस्करण वृद्धि की आवश्यकता होती है।
- शांत व्यवहार परिवर्तन को बग माना जाता है।

## गैर-लक्ष्य (जानबूझकर)

RunForge वर्तमान में निम्नलिखित करने का प्रयास नहीं करता है:

- स्वचालित रूप से मॉडल का चयन करें (उपयोगकर्ता को स्पष्ट रूप से चुनना होगा)।
- हाइपरपैरामीटर को ट्यून करें (डिफ़ॉल्ट मान प्रत्येक प्रीसेट के लिए तय हैं)।
- ऑनलाइन या वृद्धिशील प्रशिक्षण करें।
- प्रशिक्षण व्यवहार को अनुमानों के पीछे छिपाएं।

स्वचालन से अधिक शुद्धता और पारदर्शिता को प्राथमिकता दी जाती है।

## अवलोकनशीलता (v0.2.2.1+)

चरण 2.2.1 प्रशिक्षण व्यवहार को बदले बिना प्रशिक्षण रनों में दृश्यता जोड़ता है।

### रन मेटाडेटा

प्रत्येक प्रशिक्षण प्रक्रिया एक `run.json` फ़ाइल उत्पन्न करती है जिसमें निम्नलिखित जानकारी होती है:

- रन आईडी और टाइमस्टैम्प
- डेटासेट का फिंगरप्रिंट (SHA-256)
- लेबल कॉलम और फीचर की संख्या
- हटाए गए पंक्तियों की संख्या
- मेट्रिक्स का स्नैपशॉट
- आर्टिफैक्ट पाथ

### डेटासेट निरीक्षण

प्रशिक्षण शुरू करने से पहले डेटासेट का निरीक्षण करें:

```bash
python -m ml_runner inspect --dataset data.csv --label label
```

यह कॉलम के नाम, पंक्तियों की संख्या, फीचर की संख्या और लेबल सत्यापन लौटाता है।

### उत्पत्ति ट्रैकिंग

सभी रन `.ml/outputs/index.json` में अनुक्रमित हैं ताकि उन्हें ट्रैक किया जा सके:

- किसी `model.pkl` फ़ाइल के लिए, रन मेटाडेटा को वापस ट्रैक करें।
- किसी दिए गए डेटासेट फिंगरप्रिंट के लिए सभी रन खोजें।
- यह इंडेक्स केवल जोड़ने के लिए है (इसे कभी भी पुन: व्यवस्थित या हटाया नहीं जाता)।

## आर्टिफैक्ट निरीक्षण (v0.2.2.2+)

संस्करण 2.2.2 में प्रशिक्षित आर्टिफैक्ट का केवल-पढ़ने योग्य निरीक्षण जोड़ा गया है।

**निरीक्षण केवल-पढ़ने योग्य है और आर्टिफैक्ट को फिर से प्रशिक्षित या संशोधित नहीं करता है।**

### पाइपलाइन निरीक्षण

किसी `model.pkl` फ़ाइल के अंदर क्या है, इसका निरीक्षण करें बिना उसे फिर से प्रशिक्षित किए:

```bash
python -m ml_runner inspect-artifact --artifact model.pkl
```

यह संरचित JSON प्रारूप में निम्नलिखित जानकारी लौटाता है:

- पाइपलाइन चरण (क्रम में)
- चरण के प्रकार और मॉड्यूल
- प्रीप्रोसेसिंग का पता लगाना

उदाहरण आउटपुट:

```json
{
  "schema_version": "0.2.2.2",
  "artifact_path": "model.pkl",
  "pipeline_steps": [
    {"name": "scaler", "type": "StandardScaler", "module": "sklearn.preprocessing._data"},
    {"name": "clf", "type": "LogisticRegression", "module": "sklearn.linear_model._logistic"}
  ],
  "has_preprocessing": true,
  "step_count": 2
}
```

### डायग्नोस्टिक्स

संरचित डायग्नोस्टिक्स बताते हैं कि किसी रन ने जिस तरह से व्यवहार किया, उसके कारण क्या थे:

| कोड | विवरण |
|------|-------------|
| `MISSING_VALUES_DROPPED` | गायब मूल्यों के कारण पंक्तियों को हटाया गया। |
| `LABEL_NOT_FOUND` | डेटासेट में लेबल कॉलम मौजूद नहीं है। |
| `LABEL_TYPE_INVALID` | लेबल कॉलम का प्रकार अमान्य है। |
| `ZERO_ROWS` | प्रोसेसिंग के बाद डेटासेट में शून्य पंक्तियाँ हैं। |
| `ZERO_FEATURES` | डेटासेट में कोई फीचर कॉलम नहीं है। |
| `LABEL_ONLY_DATASET` | डेटासेट में केवल लेबल कॉलम है। |

सभी डायग्नोस्टिक्स मशीन-पठनीय JSON प्रारूप में हैं (लॉग पार्सिंग की आवश्यकता नहीं है)।

## रन ब्राउज़ करें (v0.2.3+)

संस्करण 2.3 में त्वरित कार्यों के साथ एक एकीकृत रन ब्राउज़र जोड़ा गया है।

## हाइपरपैरामीटर और प्रशिक्षण प्रोफाइल (v0.3.2+)

संस्करण 3.2 में स्पष्ट हाइपरपैरामीटर नियंत्रण और प्रशिक्षण प्रोफाइल जोड़े गए हैं।

### प्रशिक्षण प्रोफाइल

|---------|-------------|--------------|

| `thorough` | बेहतर गुणवत्ता के लिए अधिक पेड़/पुनरावृत्तियाँ | random_forest |

VS Code सेटिंग्स में कॉन्फ़िगर करें:
```json
{

जब प्रोफाइल और CLI दोनों पैरामीटर सेट हों:

1. **CLI `--param`** (उच्चतम प्राथमिकता)
2. **प्रोफ़ाइल-विस्तारित पैरामीटर**
3. **मॉडल डिफ़ॉल्ट** (सबसे कम प्राथमिकता)

### उत्पत्ति

हाइपरपैरामीटर और प्रोफाइल `run.json` में दर्ज किए जाते हैं:

```

जब कोई प्रोफाइल उपयोग नहीं किया जाता है, तो प्रोफाइल फ़ील्ड पूरी तरह से छोड़े जाते हैं (शून्य नहीं)।

## मॉडल-जागरूक मेट्रिक्स (v0.3.3+)

फेज 3.3 विस्तृत, मॉडल-जागरूक मेट्रिक्स जोड़ता है जिसमें क्षमता-आधारित प्रोफाइल चयन की क्षमता है।

### मेट्रिक्स प्रोफाइल

मेट्रिक्स प्रोफाइल स्वचालित रूप से मॉडल क्षमताओं के आधार पर चुने जाते हैं:

| प्रोफ़ाइल | विवरण | मेट्रिक्स |
|---------|-------------|---------|
| `classification.base.v1` | सभी क्लासिफायर | सटीकता, परिशुद्धता, रिकॉल, f1, भ्रम मैट्रिक्स |
| `classification.proba.v1` | बाइनरी + predict_proba | बेस + ROC-AUC, लॉग लॉस |
| `classification.multiclass.v1` | 3+ कक्षाएं | बेस + प्रति-कक्षा परिशुद्धता/रिकॉल/f1 |

### प्रोफ़ाइल चयन तर्क

- बाइनरी क्लासिफिकेशन + `predict_proba` → `classification.proba.v1`
- मल्टीक्लास (3+ कक्षाएं) → `classification.multiclass.v1`
- अन्यथा → `classification.base.v1`

### मॉडल क्षमताएं

| मॉडल | predict_proba | decision_function |
|-------|---------------|-------------------|
| LogisticRegression | ✅ | ✅ |
| RandomForest | ✅ | ❌ |
| LinearSVC | ❌ | ✅ (केवल ROC-AUC) |

### मेट्रिक्स आर्टिफैक्ट

अब प्रशिक्षण `metrics.v1.json` के साथ-साथ `metrics.json` उत्पन्न करता है:

```json
{
  "schema_version": "metrics.v1",
  "metrics_profile": "classification.proba.v1",
  "num_classes": 2,
  "accuracy": 0.95,
  "precision_macro": 0.94,
  "recall_macro": 0.93,
  "f1_macro": 0.94,
  "confusion_matrix": [[45, 5], [3, 47]],
  "roc_auc": 0.97,
  "log_loss": 0.15
}
```

### रन मेटाडेटा

`run.json` में अब मेट्रिक्स_v1 पॉइंटर शामिल है:

```json
{
  "schema_version": "run.v0.3.6",
  "metrics_v1": {
    "schema_version": "metrics.v1",
    "metrics_profile": "classification.proba.v1",
    "artifact_path": "metrics.v1.json"
  },
  "artifacts": {
    "model_pkl": "artifacts/model.pkl",
    "metrics_v1_json": "metrics.v1.json"
  }
}
```

### पिछला अनुकूलता

- `metrics.json` (फेज 2) अपरिवर्तित रहता है
- सभी मौजूदा उपकरण काम करना जारी रखते हैं
- `run.json` में प्रोफाइल फ़ील्ड या तो एक साथ दिखाई देते हैं या बिल्कुल नहीं।

## फ़ीचर महत्व (v0.3.4+)

फेज 3.4 समर्थित मॉडलों के लिए केवल-पढ़ने योग्य फ़ीचर महत्व निष्कर्षण जोड़ता है।

### समर्थित मॉडल

फ़ीचर महत्व केवल उन मॉडलों के लिए उपलब्ध है जिनमें देशी महत्व संकेत हैं:

| मॉडल | समर्थित | महत्व का प्रकार |
|-------|-----------|-----------------|
| RandomForest | ✅ | Gini महत्व |
| LogisticRegression | ❌ | v1 में नहीं |
| LinearSVC | ❌ | v1 में नहीं |

**कोई अनुमान नहीं**: यदि मॉडल देशी महत्व का समर्थन नहीं करता है, तो कोई आर्टिफैक्ट उत्सर्जित नहीं होता है।

### फ़ीचर महत्व आर्टिफैक्ट

RandomForest रन `artifacts/feature_importance.v1.json` उत्पन्न करते हैं:

```json
{
  "schema_version": "feature_importance.v1",
  "model_family": "random_forest",
  "importance_type": "gini_importance",
  "num_features": 10,
  "features_by_importance": [
    {"name": "feature_a", "importance": 0.35, "rank": 1},
    {"name": "feature_b", "importance": 0.25, "rank": 2}
  ],
  "features_by_original_order": [
    {"name": "feature_a", "importance": 0.35, "index": 0},
    {"name": "feature_b", "importance": 0.25, "index": 1}
  ],
  "top_k": ["feature_a", "feature_b"]
}
```

### रन मेटाडेटा

`run.json` में फ़ीचर महत्व संदर्भ शामिल है जब यह उपलब्ध हो:

```json
{
  "feature_importance_schema_version": "feature_importance.v1",
  "feature_importance_artifact": "artifacts/feature_importance.v1.json",
  "artifacts": {
    "model_pkl": "artifacts/model.pkl",
    "feature_importance_json": "artifacts/feature_importance.v1.json"
  }
}
```

जब फ़ीचर महत्व उपलब्ध नहीं होता है, तो ये फ़ील्ड पूरी तरह से छोड़े जाते हैं (शून्य नहीं)।

### डायग्नोस्टिक्स

असमर्थित मॉडल संरचित निदान उत्सर्जित करते हैं:

| कोड | विवरण |
|------|-------------|
| `FEATURE_IMPORTANCE_UNSUPPORTED_MODEL` | मॉडल देशी फ़ीचर महत्व का समर्थन नहीं करता है |
| `FEATURE_NAMES_UNAVAILABLE` | फ़ीचर नामों को हल नहीं किया जा सका |

### v1 में समर्थित नहीं

निम्नलिखित स्पष्ट रूप से v1 के दायरे से बाहर हैं:

- रैखिक मॉडलों के लिए गुणांक-आधारित महत्व
- SHAP/LIME स्पष्टीकरण
- क्रमपरिवर्तन महत्व
- आंशिक निर्भरता प्लॉट

### समर्थित हाइपरपैरामीटर

**लॉजिस्टिक रिग्रेशन:**
- `C` (फ्लोट, > 0): नियमितीकरण शक्ति
- `max_iter` (इंटीजर, > 0): अधिकतम पुनरावृत्तियाँ
- `solver` (स्ट्रिंग): अनुकूलन सॉल्वर
- `warm_start` (बूलियन): पिछले समाधान का पुन: उपयोग करें

**रैंडम फ़ॉरेस्ट:**
- `n_estimators` (int, > 0): पेड़ों की संख्या
- `max_depth` (int या None): अधिकतम पेड़ गहराई
- `min_samples_split` (int, >= 2): विभाजन के लिए न्यूनतम नमूनों की संख्या
- `min_samples_leaf` (int, > 0): प्रत्येक पत्ती के लिए न्यूनतम नमूनों की संख्या

**लीनियर एसवीसी:**
- `C` (float, > 0): नियमितीकरण की शक्ति
- `max_iter` (int, > 0): अधिकतम पुनरावृत्तियाँ

## रैखिक गुणांक (v0.3.5+)

संस्करण 3.5 में रैखिक वर्गीकरणकर्ताओं के लिए केवल-पढ़ने योग्य गुणांक निष्कर्षण जोड़ा गया है।

उन मॉडलों के लिए रैखिक गुणांक उपलब्ध हैं जिनमें `coef_` नामक एक अंतर्निहित विशेषता है:

| मॉडल | समर्थित | गुणांक का प्रकार |
|-------|-----------|------------------|
| LogisticRegression | ✅ | लॉग-ऑड्स गुणांक |
| LinearSVC | ✅ | एसवीएम गुणांक |
| RandomForest | ❌ | इसके बजाय "फ़ीचर महत्व" का उपयोग करें। |

**कोई अनुमान नहीं:** यदि मॉडल में अंतर्निहित गुणांक का समर्थन नहीं है, तो कोई भी आउटपुट फ़ाइल नहीं बनाई जाएगी।

### गुणांक स्थान (महत्वपूर्ण)

**सभी गुणांक मानकीकृत फ़ीचर स्पेस में हैं।**

इसका मतलब है:
- गुणांक उन विशेषताओं से मेल खाते हैं जो StandardScaler के बाद आती हैं।
- मान प्रति 1 मानक विचलन वृद्धि के प्रभाव को दर्शाते हैं।
- कच्चे फ़ीचर इकाइयों में स्केलिंग को "उल्टा" करने का कोई प्रयास नहीं किया जाता है।
- विभिन्न विशेषताओं के बीच गुणांकों की तुलना करना सार्थक है (समान पैमाना)।
- कच्चे फ़ीचर मानों के साथ गुणांकों की तुलना करना सार्थक नहीं है।

### रैखिक गुणांक आउटपुट फ़ाइल

रैखिक मॉडल चलाने पर `artifacts/linear_coefficients.v1.json` फ़ाइल बनती है:

```json
{
  "schema_version": "linear_coefficients.v1",
  "model_family": "logistic_regression",
  "coefficient_space": "standardized",
  "num_features": 10,
  "num_classes": 2,
  "classes": [0, 1],
  "intercepts": [
    {"class": 1, "intercept": 0.5}
  ],
  "coefficients_by_class": [
    {
      "class": 1,
      "features": [
        {"name": "feature_a", "coefficient": 2.35, "abs_coefficient": 2.35, "rank": 1},
        {"name": "feature_b", "coefficient": -1.25, "abs_coefficient": 1.25, "rank": 2}
      ]
    }
  ],
  "top_k_by_class": [
    {"class": 1, "top_features": ["feature_a", "feature_b"]}
  ]
}
```

### मल्टीक्लास समर्थन

मल्टीक्लास वर्गीकरण (3 या अधिक वर्ग) के लिए, गुणांक प्रत्येक वर्ग के अनुसार समूहीकृत होते हैं:

- प्रत्येक वर्ग के लिए अपने स्वयं के गुणांकों का एक सेट होता है।
- वर्ग लेबल नियतात्मक रूप से क्रमबद्ध होते हैं।
- v1 में वर्गों के बीच कोई एकत्रीकरण नहीं होता है।

### रन मेटाडेटा

`run.json` में रैखिक गुणांक संदर्भ शामिल है, जब यह उपलब्ध होता है:

```json
{
  "linear_coefficients_schema_version": "linear_coefficients.v1",
  "linear_coefficients_artifact": "artifacts/linear_coefficients.v1.json",
  "artifacts": {
    "model_pkl": "artifacts/model.pkl",
    "linear_coefficients_json": "artifacts/linear_coefficients.v1.json"
  }
}
```

जब गुणांक उपलब्ध नहीं होते हैं, तो ये फ़ील्ड पूरी तरह से छोड़े जाते हैं (शून्य नहीं)।

### डायग्नोस्टिक्स

असमर्थित मॉडल संरचित निदान उत्सर्जित करते हैं:

| कोड | विवरण |
|------|-------------|
| `LINEAR_COEFFICIENTS_UNSUPPORTED_MODEL` | मॉडल गुणांक निष्कर्षण का समर्थन नहीं करता है। |
| `COEFFICIENTS_MISSING_ON_ARTIFACT` | वर्गीकारक में `coef_` विशेषता नहीं है। |
| `FEATURE_NAMES_UNAVAILABLE` | फ़ीचर नामों को हल नहीं किया जा सका |

### फ़ीचर महत्व बनाम रैखिक गुणांक

| आउटपुट फ़ाइल | समर्थित मॉडल | यह क्या दिखाता है |
|----------|------------------|---------------|
| फ़ीचर महत्व (v0.3.4) | RandomForest | गिनि महत्व (ट्री-आधारित) |
| रैखिक गुणांक (v0.3.5) | लॉजिस्टिक रिग्रेशन, लीनियर एसवीसी | मॉडल गुणांक |

ये पूरक हैं:
- एन्सेम्बल मॉडलों के लिए फ़ीचर महत्व का उपयोग करें।
- व्याख्या योग्य रैखिक मॉडलों के लिए रैखिक गुणांक का उपयोग करें।

### व्याख्या मार्गदर्शिका

लॉजिस्टिक रिग्रेशन (बाइनरी) के लिए:
- सकारात्मक गुणांक: फ़ीचर में वृद्धि → सकारात्मक वर्ग की उच्च संभावना
- नकारात्मक गुणांक: फ़ीचर में वृद्धि → सकारात्मक वर्ग की कम संभावना
- परिमाण: बड़ा निरपेक्ष मान = मजबूत प्रभाव

उदाहरण: `गुणांक = 2.0` का मतलब है कि इस फ़ीचर में +1 मानक विचलन → लॉग-ऑड्स में +2.0

## व्याख्यात्मकता सूचकांक (v0.3.6+)

संस्करण 3.6 एक एकीकृत सूचकांक आउटपुट फ़ाइल जोड़ता है जो एक रन के लिए सभी व्याख्यात्मकता आउटपुट को जोड़ती है।

### उद्देश्य

व्याख्यात्मकता सूचकांक का उत्तर है: "इस रन के लिए कौन से व्याख्यात्मकता आउटपुट मौजूद हैं, वे कौन से संस्करण हैं, और वे कहां हैं?"

कोई नई गणना नहीं - केवल मौजूदा आउटपुट फ़ाइलों को लिंक और सारांशित करना।

### सूचकांक आउटपुट फ़ाइल

प्रत्येक रन के लिए `artifacts/interpretability.index.v1.json` फ़ाइल बनती है:

```json
{
  "schema_version": "interpretability.index.v1",
  "run_id": "20240101-120000-abc12345",
  "runforge_version": "0.3.6.0",
  "created_at": "2024-01-01T12:00:00+00:00",
  "available_artifacts": {
    "metrics_v1": {
      "schema_version": "metrics.v1",
      "path": "metrics.v1.json",
      "summary": {
        "metrics_profile": "classification.proba.v1",
        "accuracy": 0.95
      }
    },
    "feature_importance_v1": {
      "schema_version": "feature_importance.v1",
      "path": "artifacts/feature_importance.v1.json",
      "summary": {
        "model_family": "random_forest",
        "top_k": ["feature_a", "feature_b", "feature_c"]
      }
    },
    "linear_coefficients_v1": {
      "schema_version": "linear_coefficients.v1",
      "path": "artifacts/linear_coefficients.v1.json",
      "summary": {
        "model_family": "logistic_regression",
        "num_classes": 2,
        "top_k_by_class": [{"class": 1, "top_features": ["feat_a", "feat_b"]}]
      }
    }
  }
}
```

### उपलब्धता नियम

- अनुपलब्ध आउटपुट फ़ाइलें `available_artifacts` से **हटा दी जाती हैं** (शून्य या गलत पर सेट नहीं)।
- सूचकांक केवल तभी उपलब्धता का दावा करता है जब फ़ाइल वास्तव में मौजूद हो।
- एक न्यूनतम रन (लॉजिस्टिक रिग्रेशन) में `metrics_v1` और `linear_coefficients_v1` होंगे।
- एक रैंडम फ़ॉरेस्ट रन में `metrics_v1` और `feature_importance_v1` होंगे।

### सारांश सामग्री

सारांश में केवल संदर्भ डेटा शामिल है (कोई भी संख्यात्मक मान दोहराया नहीं गया है):

| आउटपुट फ़ाइल | सारांश में शामिल हैं: |
|----------|------------------|
| metrics_v1 | `metrics_profile`, `accuracy` (run.json फ़ाइल से) |
| feature_importance_v1 | `model_family`, `top_k` (केवल नाम, अधिकतम 5) |
| linear_coefficients_v1 | `model_family`, `num_classes`, `top_k_by_class` (केवल नाम) |

### VS कोड कमांड

"RunForge: व्यू लेटेस्ट इंटरप्रेटेबिलिटी इंडेक्स" का उपयोग करके, आप एक संरचित सारांश देख सकते हैं जिसमें व्यक्तिगत फ़ाइलों को खोलने के लिए त्वरित लिंक दिए गए हैं।
